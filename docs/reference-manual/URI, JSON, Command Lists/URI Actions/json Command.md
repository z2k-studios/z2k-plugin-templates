# json Command
The `json` command is a meta-command — instead of executing a template operation directly, it unwraps a JSON-encoded string into a full [[JSON Packages Overview|JSON Package]] and executes that. It exists to solve a practical problem: passing complex, nested command parameters through transports that flatten everything into strings, like [[URI Actions|URI query parameters]].

## Contents
- [[#Why It Exists]]
- [[#How It Works]]
- [[#Syntax]]

## Why It Exists
In a [[JSON Structure|.json file]], you can structure a command with nested objects, typed values, and clean formatting. In a URI, everything is a flat string — and encoding a JSON object as a URL parameter means percent-encoding every brace, quote, and colon. The `json` command provides an escape hatch: pass the entire command as a single `json` parameter, and let the plugin parse it on the other side.

This is especially useful for:
- Automation tools that build complex commands programmatically
- URIs where multiple field values and directives need to be passed together
- Scenarios where [[templateJsonData]] alone isn't enough — you also need to encode directives

For cases where only the field data is complex but the directives are simple, [[templateJsonData]] is usually sufficient. The `json` command is for when the entire command structure needs to travel as a single string.

## How It Works
When the plugin encounters `cmd=json`, it:
1. Reads the `json` parameter (a JSON-encoded string)
2. Parses it into a JSON object
3. Validates that the result is a plain object (not an array, primitive, or `null`)
4. Recursively calls the command processor with the parsed object, as if it had arrived from a [[JSON Structure|.json file]]

Because the parsed result is treated as a JSON source, all values preserve their types — numbers, booleans, and objects are not coerced to strings. See [[JSON Type Conversion]] for details.

> [!NOTE]
> The `json` command replaces the outer command entirely. Any other parameters alongside `cmd=json` and `json=...` are ignored — only the contents of the `json` string are processed.

## Syntax
The `json` command requires exactly one additional parameter:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `json` | string | Yes | A valid JSON string containing a complete [[JSON Packages Overview|JSON Package]]. |

The JSON string inside `json` must itself contain a `cmd` key — it's a full command, not just field data.

A Base64-encoded variant, `json64`, can be used instead of `json`. See [[JSON64 Format]].

## Examples

### In a URI
```
obsidian://z2k-templates?cmd=json&json={"cmd":"new","templatePath":"Templates/Note.md","title":"Hello","priority":3}
```

The `json` parameter value would typically be URL-encoded in practice:

```
obsidian://z2k-templates?cmd=json&json=%7B%22cmd%22%3A%22new%22%2C%22templatePath%22%3A%22Templates%2FNote.md%22%2C%22title%22%3A%22Hello%22%2C%22priority%22%3A3%7D
```

### In a JSON File
While less common (since `.json` files already support the full structure natively), the `json` command can appear in a command file:

```json
{
  "cmd": "json",
  "json": "{\"cmd\": \"new\", \"templatePath\": \"Templates/Note.md\", \"title\": \"Wrapped Command\"}"
}
```

This is primarily useful when a command file is generated by a tool that produces a stringified JSON payload rather than a nested object.

### With JSON64
To avoid URL-encoding entirely, use the Base64 variant:

```
obsidian://z2k-templates?cmd=json&json64=eyJjbWQiOiAibmV3IiwgInRlbXBsYXRlUGF0aCI6ICJUZW1wbGF0ZXMvTm90ZS5tZCIsICJ0aXRsZSI6ICJIZWxsbyJ9
```

See [[JSON64 Format]] for encoding details.
